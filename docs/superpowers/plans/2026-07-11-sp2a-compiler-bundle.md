# SP2-A — Compiler + Committed Bundle (Implementation Plan)

**Date:** 2026-07-11 · **Spec:** `docs/superpowers/specs/2026-07-10-sp2-remove-sanity-design.md` (rev 2.1)
**Lane:** SAFE (purely additive — no consumer switches, no migrations, no behavior change)
**Prereqs:** app PR #416 merged (schema `thumbnail`); courses-academy PR #1 merged after it (vendored schemas). Neither blocks Tasks 1–3 (they only affect which content fields exist).

## Load-bearing design choice

**The compiler REUSES the existing projector** (`apps/web/src/lib/content-sync/projector.ts` + friends), which already turns a courses-academy tree into the exact document shapes the sync writes to Sanity. The bundle therefore has the same shape as today's Sanity docs _by construction_ (minus `onChainStatus`/`sync`/`authoringStatus`, which the sync adds as preserve-fields, and minus Sanity's `_rev`/timestamps). This makes the parity anchor a near-trivial canonicalized compare and de-risks SP2-C (queries will return familiar shapes). Do NOT write a second projection.

Asset-path difference (deliberate): the projector rewrites `assets/x.png` → Sanity CDN upload; the compiler instead copies the file to `apps/web/public/content-assets/<courseSlug>/<lessonSlug>/x.png` and rewrites the markdown to that absolute path. The parity canonicalizer normalizes both forms (live content has zero images today, so this is exercised by fixtures, not prod data).

## Tasks

### Task 1 — `content.lock` + compile script (fetch → validate → project → emit)

**Files:**

- `apps/web/content.lock` — `{ "repo": "solanabr/courses-academy", "sha": "<40-hex>" }` (initialize at the SHA the last content sync ingested — read it from the Sanity content-sync singleton, `readContentSyncSingleton()`; record the value in the PR description).
- `apps/web/scripts/compile-content.ts` — standalone tsx script:
  1. Read `content.lock`.
  2. Fetch the tarball at that SHA with a **local ~10-line fetch** (public repo; optional `GITHUB_TOKEN` env for rate limits). Do NOT import `lib/content-sync/github.ts` (it is `server-only` + `env.server` — throws outside Next).
  3. `extractTarball` (reuse `lib/content-sync/tarball.ts` — plain module, importable).
  4. Validate every doc with `@superteam-lms/content-schema` — **fail-closed**: any violation exits non-zero with the file path + Zod issue list.
  5. Project via the existing projector into Sanity-shaped docs (no `onChainStatus`, no `sync`, no `authoringStatus`).
  6. Emit `apps/web/src/content/generated/`: one JSON module per type (`courses.json`, `lessons.json`, `instructors.json`, `achievements.json`, `quests.json`, `paths.json`) + `slots.json` (each course's `slots.lock.json`, verbatim) + `meta.json` (`{ sha, compiledAt: <from git commit of content.lock, NOT wall clock>, counts }`). Deterministic output: stable key order (`JSON.stringify` with sorted keys), newline-terminated.
- `apps/web/package.json` — `"compile-content": "tsx scripts/compile-content.ts"`.
- The generated dir is **committed** (ensure no `.gitignore` rule excludes it; add a `README.md` inside saying "generated by compile-content — do not hand-edit; CI enforces freshness").

**TDD:** projector reuse is already covered by content-sync tests; add `compile-content.test.ts` with a fixture tree (tmp dir, no network): valid tree → files emitted with expected counts + stable ordering (run twice, byte-identical); invalid doc → non-zero exit, no partial emit; missing `slots.lock.json` → error naming the course.

**Verify:** `pnpm --filter web compile-content` against the real lock SHA emits a bundle whose course/lesson counts match prod (6 courses / 76 lessons / 39 achievements / 26 quests — re-check counts at execution); commit the bundle.

### Task 2 — Asset pipeline (forward-looking; fixture-tested)

**Files:** extend `compile-content.ts`:

- Copy every `courses/<slug>/**/assets/*` file to `apps/web/public/content-assets/<mirrored path>`, rewrite `![...](assets/x.png)` in markdown blocks and `thumbnail: assets/x.png` in course docs to the public URL path.
- Reject (fail-closed) a `thumbnail`/markdown image reference whose file doesn't exist in the tree; enforce an allowlist of extensions (png/jpg/jpeg/webp/svg) and a per-file size cap (1 MB) so a content PR can't bloat the app repo.

**TDD:** fixture tree with a thumbnail + in-lesson image → files copied, paths rewritten; missing file → compile fails naming the lesson; oversized/wrong-extension → fails. Live content has zero images (measured 2026-07-11) — prod bundle must emit an empty `content-assets/`.

### Task 3 — Typed `server-only` read API (`lib/content/*`, no consumers)

**Files:**

- `apps/web/src/lib/content/store.ts` — `import "server-only"`; loads the generated JSON once (module scope), exposes typed maps: `coursesById`, `coursesBySlug`, `lessonsById`, `lessonsBySlug`, `instructorsById`, `achievementsById`, `questsById`, `pathsById`, `slotsByCourseId`. Types from `packages/types` (same interfaces `queries.ts` returns today) — zero `any`.
- NO changes to `lib/sanity/*`, NO consumer switches. The 21 query-fn re-implementations are SP2-B scope (they need the Supabase status join).

**TDD:** `store.test.ts` against a small committed test fixture (not the real bundle, so tests don't churn on content bumps): lookups resolve, unknown id → undefined, types compile strict. Plus one test asserting the module carries the `server-only` marker (import graph check or source assertion, matching however the codebase tests this elsewhere — check `lib/content-sync` patterns first).

### Task 4 — CI: freshness check + parity anchor (ordered BEFORE typecheck)

**Files:** `.github/workflows/ci.yml` (frontend job) + `apps/web/scripts/parity-check.ts`.

1. **Freshness:** run `pnpm --filter web compile-content`, then `git diff --exit-code apps/web/src/content/generated apps/web/public/content-assets` — a hand-edit or a lock/bundle drift fails CI. (Network note: fetches the public tarball; if GitHub is unreachable the step fails loudly — acceptable for CI, and deploys don't depend on it since the bundle is committed.)
2. **Parity anchor:** `parity-check.ts` reads the Sanity export (public dataset `4e3i2wwc/production`, no token) at the content-sync singleton's SHA, canonicalizes BOTH sides — strip `_rev`/`_createdAt`/`_updatedAt`/`onChainStatus`/`sync`/`authoringStatus`, sort keys, normalize asset URLs — and deep-diffs per document id. Any mismatch prints the doc id + JSON diff and fails. Runs in CI after freshness. **This anchor is temporary** (deleted with Sanity at SP2-C/D, replaced by the golden fixtures from Tasks 1–3).
3. If the singleton SHA ≠ `content.lock` SHA, parity SKIPS with a loud notice (someone synced Sanity ahead of the pin) rather than failing — the freshness check still gates correctness of the bundle itself.

**Verify:** green CI on the PR; then falsify both directions — hand-edit one generated byte (freshness must fail) and point parity at a doctored fixture (must fail). Include the falsification evidence in the task report.

Query-fn switches, Supabase `onchain_deployments`, backfill, status writers, deletions, `next.config.mjs` cleanup.

## Review gates

Per-task reviewer (spec + quality) as in SP1; whole-branch adversarial review before the PR opens; PR is SAFE-lane (additive) — normal merge flow. The bundle commit makes the PR diff large; reviewers review the compiler + a spot-sample of the bundle, with the freshness check as the mechanical guarantee for the rest.
